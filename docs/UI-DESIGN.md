# Zhihu Reader UI 设计规范

| 项目 | 内容 |
| --- | --- |
| 状态 | 已确认 |
| 选定方案 | A — 专注阅读 |
| 适用界面 | Zhihu Reader 独立 Obsidian View |
| 决策日期 | 2026-07-14 |

## 1. 设计决策

正式 UI 采用方案 A：居中单列的专注阅读器。

选择该方案的原因：

- 与“一次只阅读一篇回答”的产品原则一致。
- 视觉结构接近 Obsidian 原生阅读视图，不会像另一个网页应用。
- 问题上下文、回答正文和翻页操作具有清晰的纵向阅读顺序。
- 在窄侧栏、主工作区和移动端之间更容易保持一致。
- 历史、保存和设置保持可达，但不会与回答正文争夺注意力。

不采用左右分栏和沉浸翻页作为正式布局。正式产品不提供布局切换设置。

## 2. 整体结构

```text
┌──────────────────────────────────────────────────────────┐
│ 知 Zhihu Reader   打开链接  每日热榜  历史列表  刷新  登录 │
│          ← 上一回答   第 2 篇   综合排序   下一回答 →      │ sticky
├──────────────────────────────────────────────────────────┤
│                                                          │
│       ┌──────────────────────────────────────────┐       │
│       │ 问题 · 128 个回答 · 2.4 万关注           │       │
│       │ 为什么……？                              │       │
│       │ 问题描述、话题、展开/收起                │       │
│       └──────────────────────────────────────────┘       │
│                                                          │
│       ┌──────────────────────────────────────────┐       │
│       │ 作者信息                                  │       │
│       ├──────────────────────────────────────────┤       │
│       │                                          │       │
│       │ Obsidian 原生 Markdown 回答正文           │       │
│       │                                          │       │
│       ├──────────────────────────────────────────┤       │
│       │ 原文 · 发布时间              保存回答     │       │
│       └──────────────────────────────────────────┘       │
└──────────────────────────────────────────────────────────┘
```

阅读内容最大宽度为 `790px`，水平居中。宽屏增加两侧留白，不拉宽正文；窄屏改为 `12–18px` 页面边距。

## 3. 组件规范

### 3.1 ReaderToolbar

吸附在 View 顶部，使用轻微半透明背景和底部分隔线。空白、加载和错误状态只显示约 `50px` 的全局操作行；回答就绪后增加紧凑的回答导航行。

左侧：

- “知”字插件标识
- `Zhihu Reader`
- 小号状态文字，例如“专注阅读”

右侧按以下顺序排列：

1. 打开链接
2. 每日热榜
3. 历史列表
4. 刷新
5. 登录状态或用户头像

回答就绪时，第二行居中显示 `AnswerToolbarNavigation`：上一回答、当前位置、排序方式和下一回答。它随工具栏保持可见，正文底部不再重复显示导航。

插件设置通过 Obsidian 原生设置页管理。若工具栏保留设置入口，应直接打开该插件的设置页，不使用自定义设置弹窗。

### 3.2 QuestionSummary

- 使用 `var(--background-primary-alt)` 或 `var(--background-secondary)` 与回答卡片区分。
- 显示问题标题、描述、话题、回答数和关注数。
- 问题标题使用比正文更大的 Obsidian 标题字号，最多不限制行数。
- 问题描述默认根据设置决定展开或折叠，切换回答时保持当前折叠状态。
- 展开/收起操作位于摘要底部右侧。

### 3.3 AnswerCard

回答卡片是页面的主要视觉对象：

- 背景使用 `var(--background-primary)`。
- 使用 1px Obsidian 边框、`var(--radius-l)` 圆角和轻量阴影。
- 桌面端内边距 `24px 32px`；窄屏内边距 `18px`。
- 顶部显示作者头像、姓名、简介、赞同数和评论数。
- 问题 URL 与回答 URL 使用相同的回答卡片，不显示入口类型或特殊回答徽标。
- 正文必须由 `MarkdownRenderer` 渲染，不直接插入知乎 HTML。
- 正文字号遵循 Obsidian `--font-text-size`，行高约为 `1.85–1.95`。
- 底部显示发布时间、在浏览器打开和保存回答。

保存状态：

- 未保存：次要按钮“保存回答”。
- 保存中：禁用按钮并显示进度。
- 已保存：使用强调色弱背景显示“已保存”，点击后打开笔记。
- 文件已存在：提示覆盖或打开现有文件，不能静默覆盖。

### 3.4 AnswerToolbarNavigation

位于粘性工具栏第二行，不覆盖正文：

- 左侧按钮：上一回答。
- 中间：当前位置和排序说明。
- 右侧按钮：下一回答，作为更强的主操作。
- 第一个队列项禁用上一回答；没有下一页时禁用下一回答。
- 加载下一批时保留当前回答，下一回答按钮显示加载状态。
- 下一页失败与“已是最后一篇”使用紧凑状态文案显示在同一行；失败状态提供重试。

位置文案对两类 URL 完全一致：统一使用 `第 3 篇`，并显示“综合排序”或“最近更新”。回答 URL 只在数据层将 URL 对应回答放到第一篇，不在视觉和操作层做特殊标记。

### 3.5 HistoryPopover

点击工具栏“历史列表”后，以按钮为锚点在其下方显示：

- 容器尺寸、边框、圆角、阴影、标题栏间距和滚动高度与 DailyHotPopover 完全一致：桌面端宽度约 `460px`，最大高度约 `600px`。
- 顶部沿用 DailyHotPopover 的结构：左侧显示“查询历史”和问题数量，右侧“清空”按钮对应热榜的“刷新”按钮。
- 按最后查询时间倒序显示问题标题和弱化的相对查询时间；今天和昨天显示具体时间，较早记录显示日期，悬停可查看完整时间。
- 列表内边距、行高、标题字号、两行截断和悬停背景与热榜条目一致；标题按钮必须完全重置主题按钮边框、阴影和字体样式。
- 点击标题后关闭 Popover，以问题模式打开对应问题。
- 条目右侧删除按钮叠放在整行点击区域内，默认弱化，悬停或聚焦时增强，不将条目拆成两个视觉按钮。
- “清空”执行前二次确认；面板不增加与热榜不同的独立底栏。
- 点击外部或按 Escape 关闭，不改变当前回答。
- 空状态的留白、颜色和对齐方式与热榜空状态一致。

Popover 不显示回答 ID、作者、回答位置或回答预览。

### 3.6 DailyHotPopover

点击工具栏“每日热榜”后，以按钮为锚点在其下方显示：

- 桌面端宽度建议 `420–460px`，最大高度约 `600px`，超出后内部滚动。
- 顶部显示“每日热榜”、最近更新时间和刷新按钮。
- 每条使用无边框列表行，依次显示名次、问题标题、热度文案和回答数；摘要与缩略图仅在数据存在时展示。
- 前三名使用强调色和字重共同突出，不能只依靠颜色表达排名。
- 点击条目后关闭 Popover，并以问题模式打开对应问题。
- 加载时保留已有缓存列表；刷新失败时在列表上方显示错误，不清空仍可用内容。
- 未登录且接口拒绝访问时，显示“请先在插件设置中登录”和重试入口。
- 点击外部或按 Escape 关闭，不改变当前回答；与 HistoryPopover 不同时打开。

### 3.7 ZhihuUrlModal

通过命令“打开知乎内容”触发：

- 标题：“打开知乎问题或回答”。
- 单一 URL 输入框。
- 显示支持的问题 URL 和回答 URL 示例。
- 主按钮：“打开阅读器”。
- 次按钮：“取消”。
- Enter 提交，Escape 关闭。
- URL 无效时保留输入并在输入框下方显示错误。

### 3.8 空白阅读器

通过 Ribbon 打开但尚未查询内容时，使用方案 A 的居中空状态：

- “知”字标识
- 标题 `Zhihu Reader`
- 一句说明：“输入知乎问题或回答链接开始阅读”
- 主按钮“打开链接”
- 次要入口“从剪贴板打开”
- 最近问题仅通过工具栏历史 Popover 查看，不在空状态铺开列表。

## 4. 页面状态

### 4.1 问题查询

1. 显示问题摘要骨架和回答骨架。
2. feed 返回后渲染第一篇回答。
3. 内存保存本批回答，但 DOM 中只挂载当前回答。

### 4.2 回答查询

1. 获取 URL 对应回答及完整问题摘要，将该回答显示为第一篇。
2. 第一篇显示后，后台加载默认 6 篇 feed 回答。
3. 后台加载期间下一回答按钮显示加载状态。
4. 后台失败不遮挡第一篇回答，在导航区提供重试。

### 4.3 错误

- 问题加载失败：在问题摘要位置显示错误卡片、重试和浏览器打开。
- 当前回答转换失败：保留作者信息，正文显示转换失败占位符。
- 下一批加载失败：当前回答保持可读，仅工具栏回答导航显示错误。
- 登录失效：工具栏显示未登录状态，公开内容继续匿名读取。

## 5. 响应式规则

### 大于 900px

- 内容宽度固定为最大 `790px`。
- 工具栏显示图标和文字。
- 问题和回答卡片使用完整内边距。

### 600–900px

- 内容宽度为 View 宽度减 `48px`。
- 适当压缩工具栏间距。
- 作者统计可以缩短文案。

### 小于 600px

- 页面边距 `12px`。
- 工具栏次要操作只保留图标和可访问名称。
- 作者赞同/评论统计可移到作者信息下方。
- 回答导航允许在工具栏第二行内部换行，上一回答和下一回答仍保留文字。
- HistoryPopover 宽度不超过视口减 `24px`。
- DailyHotPopover 宽度不超过视口减 `24px`，缩略图可隐藏以保证标题可读。

任何宽度下都不切换为左右分栏，也不同时展示多篇回答。

## 6. 视觉语言

- 只使用普通 CSS 和 Obsidian CSS Variables。
- 插件主强调色继承 `var(--interactive-accent)`，不固定使用知乎蓝。
- 背景、边框、正文、弱文字分别使用 Obsidian 主题变量。
- 正文优先使用 Vault 配置的文本字体；工具栏使用界面字体。
- 动画时长控制在 `120–180ms`，并尊重 `prefers-reduced-motion`。
- 阴影只用于回答卡片、Popover 和 Modal，不用于所有容器。

## 7. 无障碍

- 所有图标按钮必须有 `aria-label` 和 tooltip。
- HistoryPopover 使用焦点管理：打开后焦点进入列表，关闭后返回“历史列表”按钮。
- DailyHotPopover 使用相同的焦点管理；刷新、重试和榜单条目均可由键盘访问。
- URL Modal 打开后自动聚焦输入框并形成焦点陷阱。
- 加载、错误和保存结果通过可感知状态区域通知。
- 上一回答/下一回答支持键盘操作，但不得抢占输入框中的方向键。
- 颜色不能成为区分错误或保存状态的唯一方式。

## 8. 正式组件映射

建议将方案 A 重写为以下正式组件，不直接复制原型代码：

```text
ZhihuAnswersView
└── ZhihuAnswersApp
    ├── ReaderToolbar
    │   ├── AnswerToolbarNavigation
    │   ├── DailyHotPopover
    │   └── HistoryPopover
    ├── EmptyReaderState
    ├── QuestionSummary
    ├── AnswerCard
    │   ├── AnswerAuthor
    │   └── AnswerActions
    └── ZhihuUrlModal
```

组件只接收经过 Zod 校验并转换后的领域数据。API 请求、阅读队列、历史持久化和 Vault 写入不放入展示组件。
