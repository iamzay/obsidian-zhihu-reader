# Zhihu Reader 产品需求文档

| 项目 | 内容 |
| --- | --- |
| 产品名称 | Zhihu Reader |
| 产品形态 | Obsidian 社区插件 |
| 文档状态 | Draft v1.6 |
| 目标版本 | MVP → 1.0 |
| 核心定位 | 将知乎内容自然接入 Obsidian 的阅读、保存和知识沉淀工作流 |

## 1. 产品概述

Zhihu Reader 不是把知乎 App 搬进 Obsidian，也不试图提供推荐流、社交互动或内容发布。它解决的是一个更窄但完整的问题：用户遇到值得阅读的知乎问题或回答后，能在 Obsidian 内舒适地临时阅读，只把真正有价值的内容转成可维护、可链接、可搜索的 Markdown 笔记。

产品主链路为：

```text
调用命令 → 输入知乎链接 → 在独立阅读器中临时阅读 → 保存有价值的回答 → 在 Obsidian 中继续整理和链接
```

### 1.1 产品原则

1. **默认不写入 Vault**：打开、翻页和临时阅读只使用内存或插件缓存，不创建 Markdown 文件。
2. **用户明确决定后再沉淀**：只有点击“保存回答”才写入 Vault。
3. **产物属于用户**：保存结果是普通 Markdown，不依赖插件也能阅读和维护。
4. **原生而非网页套壳**：使用 Obsidian View、Markdown 渲染器、主题变量和文件 API，不使用知乎网页 WebView 作为阅读界面。
5. **可追溯但不过度侵入**：笔记保留来源、作者、抓取时间和内容 ID，同时为用户编辑预留稳定区域。
6. **一次专注一篇回答**：阅读面板每屏只展示一篇回答，通过按钮切换，后续答案按需分页获取。
7. **历史不等于沉淀**：阅读历史只按问题 ID 记录用户查询过的问题，保存在插件数据中，不创建 Vault 文件。

## 2. 目标用户与场景

### 2.1 目标用户

- 使用 Obsidian 建立个人知识库，经常从知乎获取概念解释、经验总结或观点材料的用户。
- 希望先阅读再筛选，不愿让剪藏内容大量污染 Vault 的用户。
- 需要保留原始出处，同时会在原文基础上补充自己的理解、双链和标签的用户。

### 2.2 核心场景

#### 场景 A：临时阅读

用户调用“打开知乎内容”命令，插件弹出问题/回答输入框。用户粘贴知乎 URL 后，插件识别链接并在独立阅读面板中展示内容。问题链接默认显示 feed 中的第一篇回答；回答链接将 URL 对应的回答放在队列第一篇，并在后台加载同一问题的回答。两种入口使用完全一致的阅读界面、位置文案和上一回答/下一回答交互。用户看完后直接关闭，不在 Vault 中留下文件。

#### 场景 B：保存完整回答

用户认为某篇回答值得长期保留，点击“保存回答”。插件把回答转换为 Markdown，写入配置目录，并显示可点击的目标文件。默认一篇回答对应一个文件。

#### 场景 C：从每日热榜发现问题

用户在阅读器工具栏或命令面板打开“每日热榜”，通过轻量 Popover 浏览当前热门问题。点击条目后直接进入该问题的统一阅读页；只浏览榜单不会写入 Vault，也不会产生查询历史。

## 3. 范围

### 3.1 MVP 范围

- 调用命令后弹出输入框，识别用户输入或从剪贴板粘贴的知乎问题、回答链接。
- 在独立 Obsidian View 中展示问题和回答。
- 问题及回答 HTML 转换为 Markdown，并由 Obsidian 原生 Markdown 渲染器展示。
- 每屏只展示一篇回答；问题链接和回答链接都通过上一回答/下一回答按钮切换，并在需要时分页获取。
- 回答链接将 URL 对应的回答放在当前阅读队列首项，同时后台加载同一问题的回答。
- 知乎登录、登录状态展示和退出。
- 将单篇回答保存为 Markdown 文件。
- 记录和管理用户查询过的问题历史；查询回答 URL 时记录该回答所属的问题。
- 在阅读器内查看每日热榜并进入对应问题阅读页。
- 插件设置、错误状态、重试、空状态和已保存状态。

### 3.2 1.0 完善范围

- 图片本地化、附件目录与命名策略。
- 快捷键、移动端适配、无障碍和长回答性能优化。
- 可配置笔记模板和问题索引笔记。

### 3.3 非目标

- 首页推荐流、关注动态和搜索全站内容。
- 点赞、评论、收藏、关注和通知。
- 发布或编辑知乎回答。
- 摘录到当前笔记或其他已有笔记。
- 自动检测原文变更或更新已保存笔记。
- 连续滚动或同屏展示多篇回答。
- 完整复刻知乎网页样式、广告或商业模块。
- 绕过知乎的付费、权限、风控或反爬限制。

## 4. 信息架构与界面

### 4.1 入口

标准打开流程必须固定为：

```text
调用命令 → 弹出问题/回答 URL 输入框 → 用户输入或粘贴知乎 URL → 校验 → 打开阅读面板
```

插件提供以下入口：

- 命令面板：`Zhihu Reader: 打开知乎内容`，调用后弹出 URL 输入框。
- 左侧 Ribbon 图标：打开空白阅读器或聚焦现有阅读器。
- 命令面板：`Zhihu Reader: 从剪贴板打开`，读取剪贴板中的知乎 URL，若合法则跳过用户输入，直接打开阅读面板
- 命令面板：`Zhihu Reader: 查看历史列表`，聚焦阅读器并打开与工具栏相同的历史 Popover。
- 命令面板：`Zhihu Reader: 查看每日热榜`，聚焦阅读器并打开与工具栏相同的热榜 Popover。
- 命令面板：`Zhihu Reader: 保存当前回答`，仅在阅读面板有活动回答时可用。

输入框标题为“打开知乎问题或回答”，占位提示展示受支持 URL 示例，并提供“打开”和“取消”。按 Enter 提交，按 Escape 关闭。插件不持续监听剪贴板；只有用户触发剪贴板命令时才读取，以避免隐私和权限问题。

### 4.2 阅读器布局

UI 已确认采用方案 A“专注阅读”：顶部粘性工具栏集中放置全局操作和回答导航，内容区使用最大宽度约 `790px` 的居中单列。问题摘要与回答卡片按阅读顺序纵向排列；不提供左右分栏或布局切换。详细规范见 [UI 设计规范](UI-DESIGN.md)。

桌面端采用三段式阅读器布局：

1. **顶部工具栏**
   - 打开链接
   - 每日热榜
   - 历史列表
   - 刷新
   - 登录状态
   - 第二行回答导航：“上一回答”、答案序号、排序方式和“下一回答”
2. **问题区**
   - 标题、问题描述、话题
   - 回答数、关注数等只读元信息
   - 折叠/展开问题描述
3. **回答阅读区**
   - 作者、简介、发布时间、赞同数、评论数
   - Markdown 正文
   - 保存、在浏览器打开

阅读面板始终使用单篇模式：无论视图宽度如何，每屏只展示和渲染一篇回答，不提供连续滚动模式或回答列表侧栏。问题标题和描述位于回答上方，可独立折叠；切换回答时保留问题区状态，并将回答区滚动位置重置到顶部。

问题 URL 与回答 URL 的阅读面板和交互保持一致，只有队列首项的来源不同：

| 输入类型 | 阅读面板行为 |
| --- | --- |
| 问题 URL | 展示问题信息和当前回答；允许通过按钮在多个回答间切换 |
| 回答 URL | 将 URL 对应回答作为首项，同时后台加载同一问题的回答；使用与问题 URL 相同的按钮、序号和排序交互 |

### 4.3 历史列表 Popover

- 用户点击工具栏“历史列表”按钮后，在按钮下方打开 Popover，不离开当前阅读内容。
- Popover 按最后查询时间倒序展示历史问题，每个条目的主要内容为最多两行的问题标题，查询时间作为弱化辅助信息。
- 点击问题标题后关闭 Popover，并以问题 URL 模式打开该问题阅读页，从第一批回答开始阅读。
- 每个条目提供弱化的删除操作；Popover 标题栏提供“清空”，执行清空前需要确认。
- 历史条目较多时 Popover 使用固定最大高度并在内部滚动，不扩张阅读器布局。
- 点击 Popover 外部或按 Escape 关闭；关闭不会改变当前阅读内容。
- 没有历史时显示与每日热榜一致的简洁空状态，不展示空白列表。

### 4.4 每日热榜 Popover

- 用户点击工具栏“每日热榜”按钮后，在按钮下方打开 Popover，不离开当前阅读内容。
- 榜单按知乎当前排序展示最多 50 个问题，条目包括名次、问题标题、热度文案、回答数，以及可用时的问题摘要和缩略图。
- 点击热榜条目后关闭 Popover，并以问题 URL 模式打开对应问题阅读页；成功打开后按正常查询规则记录问题历史。
- 仅打开、刷新或浏览热榜不产生查询历史，不写入 Vault，也不创建 Markdown 文件。
- 热榜在内存中缓存 5 分钟；缓存有效时再次打开直接展示，用户仍可点击刷新强制重新请求。
- Popover 提供加载、空、失败和重试状态。接口要求登录时，引导用户前往插件设置登录，不尝试绕过知乎权限。
- 每日热榜与历史列表 Popover 互斥；点击外部或按 Escape 关闭，并将焦点返回触发按钮。

### 4.5 页面状态

阅读器必须覆盖以下状态：

- 首次使用：说明可粘贴的链接类型和登录价值。
- 无链接空状态。
- 解析中、请求中和正文转换中。
- 登录状态。
- 无回答、回答被删除或仅作者可见。
- 网络错误、限流、API 结构变化和内容转换失败。
- 保存状态。
- 热榜加载、空列表、登录要求和请求失败状态。

错误状态应保留当前已加载内容，提供重试和在浏览器打开，不用空白页替换整个阅读器。

## 5. 功能需求

### FR-01 命令打开与链接识别

- 用户调用 `Zhihu Reader: 打开知乎内容` 后，插件必须先显示 URL 输入框，不直接打开阅读面板。
- 用户提交有效 URL 后才创建或聚焦 Zhihu Reader View，并加载目标内容。
- 若阅读面板已经打开，新目标复用该 View 并重置旧的回答导航状态。
- 用户取消输入时不打开 View、不发起请求，也不修改当前阅读内容。

支持识别：

- `https://www.zhihu.com/question/{questionId}`
- `https://www.zhihu.com/question/{questionId}/answer/{answerId}`
- 包含查询参数、锚点或尾部斜杠的上述链接
- 知乎短链接，经过用户确认后解析跳转目标

识别后生成标准化目标：

```ts
type ZhihuTarget =
  | { type: "question"; questionId: string }
  | { type: "answer"; questionId?: string; answerId: string };
```

无效链接不发起网络请求，错误直接显示在输入框下方，输入内容保持不变。剪贴板命令若发现多段文本，则提取第一个受支持 URL 预填输入框并等待用户确认。

### FR-02 临时阅读与缓存

- 打开内容不会创建 Vault 文件、目录、标签或附件。
- 当前会话数据优先保存在内存中。
- 可选的响应缓存写入插件数据目录，而非用户笔记目录。
- 缓存有容量和过期时间限制，可在设置中清空或关闭。
- 关闭 View 后结束当前阅读会话

### FR-03 问题和回答获取

- **问题 URL**：获取问题元数据和第一批回答，以 feed 中第一篇作为阅读队列首项。
- **回答 URL**：获取 URL 对应回答，并使用其问题 ID 加载完整问题元数据；问题摘要中的回答数、关注数等信息必须与问题 URL 一致。
- URL 对应回答作为队列首项，后台请求同一问题的回答 feed，组成 `[URL 对应回答, ...feed 回答]`。
- URL 对应回答不计入 feed 的 `limit`。默认设置下，回答 URL 的初始队列最多包含该回答和 6 篇 feed 回答。
- 问题回答 feed 每次请求的回答数可配置，默认 6，对应请求参数 `limit=6`；允许范围为 1–20。
- feed 返回队列首项时按回答 ID 去重；去重集合覆盖当前会话的所有分页，避免同一回答再次出现。
- 两种 URL 都只通过“上一回答”“下一回答”按钮导航，不通过页面滚动触发回答切换。
- 用户点击“下一回答”时，优先切换到当前队列中尚未展示的下一篇；当前 feed 批次用完后，才请求 `paging.next` 指向的下一批回答。
- 已经阅读过的回答保存在当前会话内存中，以支持“上一回答”；无论队列数量多少，UI 同时只挂载一个回答正文。
- 使用 API 返回的 `paging.next` 作为下一页游标，不自行推算 offset。
- 以回答 ID 去重，并防止连续点击造成重复并发请求。
- 支持 `default` 和 `updated` 排序。由回答 URL 进入时，切换排序只重新加载首项之后的 feed 回答，URL 对应回答保持在队列首项。
- 回答 URL 的 feed 请求失败时不影响首项回答阅读；工具栏回答导航显示加载失败和重试。问题 URL 的首批 feed 请求失败时显示问题级错误。
- 回答 URL 的 feed 回答尚未加载完成时，“下一回答”显示加载状态且不可重复触发请求；加载成功后自动启用。
- 不在滚动时加载下一批回答。当前批次用尽后，由下一次“下一回答”操作触发分页请求。
- 队列首项禁用“上一回答”；没有更多回答时禁用“下一回答”并显示“已是最后一篇”。

### FR-04 原生 Markdown 阅读

处理管线：

```text
API JSON → Zod 校验 → 提取安全 HTML → Turndown → Markdown 后处理 → Obsidian MarkdownRenderer
```

转换至少支持：

- 标题、段落、粗体、斜体、删除线
- 有序/无序列表、引用、分割线
- 链接和图片
- 行内代码与代码块
- 表格
- 知乎公式到 Obsidian/MathJax 兼容语法
- 脚注和回答内锚点

必须移除脚本、样式、事件属性、追踪元素和不安全协议。无法转换的内容显示明确占位符，并提供“在浏览器查看原文”。

渲染使用当前 Vault 的主题、字体、代码高亮和 CSS Variables。插件 CSS 只负责阅读器布局，不复制知乎页面 CSS。

### FR-05 保存回答

- 每个回答卡片提供主操作“保存回答”，保存目标始终是当前显示的回答，而不是回答 URL 对应的首项回答。
- 默认一篇回答保存为一个文件。
- 默认路径：`Zhihu Reader/{问题标题}/{作者名} - {回答ID}.md`。
- 路径和文件名模板可配置，所有动态字段必须经过文件名清理。
- 保存前检查 `zhihu_answer_id`，若存在提示是否覆盖
- 保存成功后按钮变为“打开笔记”（在新标签页打开）
- 文件写入失败时不留下半成品；图片下载失败不阻断正文保存，但需列出失败项。

默认笔记结构：

```markdown
---
type: zhihu-answer
zhihu_answer_id: "123456789"
zhihu_question_id: "987654321"
question: 问题标题
author: 作者名
source: https://www.zhihu.com/question/987654321/answer/123456789
created_at: 2026-01-01
---

# 问题标题

> [!info] 回答信息
> 作者：[作者名](作者主页链接) · 1234 赞同
> [查看知乎原文](来源链接)

回答正文

## 我的笔记

```

保存后的文件完全归用户维护。插件不检测线上变更，也不更新或覆盖已保存文件。

### FR-06 图片和附件

- 临时阅读默认显示远程图片，不写入 Vault。
- 用户可设置：图片保留远程链接或下载到 Vault
- 下载默认遵循 Obsidian 附件目录设置，也可使用独立目录模板。
- 对图片按内容 URL/哈希去重，清理跟踪参数并保留必要尺寸信息。
- 图片文件名冲突时使用稳定后缀，不覆盖无关附件。

### FR-07 登录

- 匿名状态可以读取公开内容。
- 设置页提供“登录知乎”，优先采用二维码登录：插件展示二维码，用户使用知乎 App 扫码，插件轮询确认结果。
- 登录成功后展示头像/昵称、登录时间和“退出登录”。
- Cookie 仅保存在插件私有数据中，不写入 Markdown、日志或剪贴板。
- 日志必须脱敏 `Cookie`、`d_c0`、token 和 ZSE 相关值。
- 登录失效时自动降级为匿名阅读并提示重新登录，不删除已经加载或保存的内容。
- 退出登录清除会话和认证缓存，不删除笔记或普通阅读缓存。

### FR-08 插件设置

设置按以下分组展示：

#### 账号

- 登录状态、登录、退出

#### 阅读

- 默认排序：综合 / 最近更新
- 问题回答 feed 每次请求的回答数：1–20，默认 6
- 最大历史条数，默认 50

#### 保存

- 笔记目录，默认 `Zhihu Reader`
- 文件路径模板
- Frontmatter 字段和别名
- 保存后自动打开笔记
- 是否下载图片到 Vault
- 下载图片时的附件位置

### FR-09 阅读历史（问题查询记录）

- 历史始终以问题为单位，唯一内容标识只保存问题 ID，不保存回答 ID 或“问题/回答”目标类型。
- 用户查询问题 URL 并成功加载后，记录该问题 ID 和问题标题。
- 用户查询回答 URL 并成功加载后，从回答数据中取得所属问题 ID 和问题标题，记录该问题，而不是记录回答。
- 用户在阅读面板中切换回答不新增历史，也不改变历史数据结构。
- 同一问题下无论查询过多少个回答，都只保留一条历史；再次查询时更新最后查询时间并移动到列表顶部。
- 历史项包含问题 ID、用于列表展示的问题标题和最后查询时间；问题 URL 由问题 ID 生成，不单独存储。不保存回答 ID、作者、回答正文、Cookie 或请求响应。
- 阅读历史保存在插件数据目录，不写入 Vault，不创建 Markdown 文件。
- 工具栏“历史列表”按钮使用 Popover 展示历史，每个条目显示问题标题。
- 点击历史条目后以问题 URL 模式打开问题阅读页，从第一批回答开始，不恢复此前的回答位置。
- 历史条数超过设置上限时自动删除最旧记录；上限默认 50。
- 查询历史始终记录；用户可在历史 Popover 中删除单条或确认后清空。
- 删除历史只删除问题查询记录，不删除已保存的 Markdown 笔记、附件或缓存。

### FR-10 每日热榜

- 通过 `ZhihuGateway.getHotList()` 获取知乎当前热榜；React 展示组件不得直接拼接 URL、读取 Cookie 或解析原始 JSON。
- 默认请求 50 条，领域数据中的问题 ID 始终使用字符串。解析时优先使用能保留精度的字符串标识，不接受发生精度损失的数字 ID。
- 热榜状态由独立模块管理，请求进行中合并重复调用，并忽略视图销毁后返回的过期结果。
- 成功数据在当前插件进程内缓存 5 分钟；普通打开复用有效缓存，显式刷新绕过缓存。
- 点击条目以问题模式复用现有阅读流程；只有问题成功加载后才写入查询历史。
- 浏览、刷新、加载失败或关闭热榜均不写查询历史和 Vault。
- 登录不足、网络错误、限流和响应结构变化均在 Popover 内给出可恢复提示；已有热榜数据和当前回答不得被错误状态清空。

## 6. 数据与状态

### 6.1 插件设置数据

保存在 Obsidian 插件数据文件中，包括设置、认证会话、阅读历史和缓存索引。认证信息不得进入 Vault Markdown。

### 6.2 阅读历史数据

```ts
interface ReadingHistoryEntry {
  questionId: string;
  questionTitle: string;
  lastQueriedAt: string;
}
```

历史记录使用 `questionId` 作为唯一键，时间使用 ISO 8601 字符串。打开历史条目时，根据问题 ID 生成 `https://www.zhihu.com/question/{questionId}` 形式的 canonical 问题 URL。

### 6.3 Vault 数据

仅包含用户主动创建的：

- 回答 Markdown 文件
- 可选下载图片和附件

### 6.4 稳定标识

使用字符串保存知乎问题和回答 ID，避免 JavaScript 对超大整数精度不足。文件匹配以 `zhihu_answer_id` 为主键，路径不是身份。

## 7. 非功能需求

### 7.1 性能

- 命令输入框应在 200 ms 内出现，不等待网络。
- 问题 URL 每次获取设置数量的一批回答，默认 6；回答 URL 将 URL 对应回答置于首项，并在后台获取同样数量的 feed 回答。
- 回答 URL 在完整问题摘要和首项回答返回后显示内容，后台 feed 不阻塞首项阅读。
- 长回答渲染不阻塞主线程超过可感知时长；转换和图片处理需要分阶段执行。
- 阅读面板同时只挂载一个回答正文；切换后及时卸载上一回答的渲染节点。
- 相同 URL 的进行中请求应合并，并支持在切换目标时取消过期请求。
- 查询历史写入不能阻塞查询结果渲染；超出上限后的裁剪应与本次历史写入一并完成。

### 7.2 可靠性

- API 响应必须先经过 Zod 边界校验。
- 网络请求支持超时、取消和有限次数退避重试；不对 401/403 盲目重试。
- 分页失败不影响当前回答阅读，上一回答仍可切换，并提供下一页重试。
- 保存采用先生成、后落盘的原子化思路，避免损坏现有文件。
- 查询历史写入失败不影响当前内容阅读，并在下一次成功写入时重试持久化。
- API 字段缺失时尽量降级展示，并在脱敏日志中记录 schema 问题。

### 7.3 安全与隐私

- 不收集遥测，不上传 Vault 内容。
- 不记录完整 Cookie 或回答私密内容。
- 阅读历史包含用户查询过的问题 ID 和标题，用户可以在历史 Popover 中删除或清空。
- 所有远程内容视为不可信输入，转换和渲染前必须净化。
- 不在插件内收集知乎密码。
- 清除插件数据前需要说明影响并二次确认。

### 7.4 兼容性和可访问性

- 支持 Obsidian 桌面端；移动端在 API 和剪贴板能力可用时提供核心阅读与保存能力。
- 兼容浅色/深色主题及常见社区主题，不硬编码背景和正文颜色。
- 所有主要操作可通过键盘完成，焦点状态清晰。
- 图标按钮有可访问名称，加载和错误状态可被屏幕阅读器感知。
- 尊重系统减少动画设置。

## 8. 核心流程验收标准

### AC-01 通过命令打开

1. 用户执行 `Zhihu Reader: 打开知乎内容`。
2. 插件弹出“打开知乎问题或回答”输入框，尚未打开阅读面板或发起网络请求。
3. 用户输入有效知乎 URL 并确认。
4. 插件打开独立 View 并加载目标；临时阅读过程中 Vault 没有新文件或目录。

### AC-02 阅读问题

1. 用户输入问题 URL。
2. 阅读面板展示问题信息和第一篇回答。
3. 页面只挂载并显示一篇回答正文，不出现回答连续列表。
4. 用户只能通过“上一回答”“下一回答”按钮切换回答。

### AC-03 阅读回答链接

1. 用户输入回答 URL。
2. 阅读面板加载完整问题上下文，将 URL 对应回答显示为第一篇。
3. 插件后台请求默认 6 篇 feed 回答，将 URL 对应回答保持在队列首项，并按回答 ID 去重。
4. 页面使用和问题 URL 完全一致的“第 1 篇”“上一回答”“下一回答”文案与交互，不显示特殊回答徽标或返回入口。
5. 后台 feed 失败时首项回答仍可阅读，并可从导航区重试。

### AC-04 按钮切换与分页

1. 问题 URL 或回答 URL 的当前回答加载完成。
2. 首次 feed 请求使用设置中的回答数；默认情况下请求参数为 `limit=6`。
3. 用户点击“下一回答”时，插件优先切换当前批次中的下一篇，不发起新请求。
4. 当前批次最后一篇之后再次点击“下一回答”，插件才请求 `paging.next`；默认配置下该批次包含 6 篇。新回答按 ID 去重，请求失败时当前回答保持可读并可重试。

### AC-05 保存回答

1. 用户点击当前回答的“保存回答”。
2. 插件创建一个带稳定 frontmatter 和来源的 Markdown 文件。
3. 再次保存同一回答时提示文件已存在，是否覆盖
4. 文件在停用插件后仍能被 Obsidian 正常阅读。

### AC-06 登录失效

1. 已保存会话过期。
2. API 返回认证失败。
3. 插件清楚提示未登录状态并降级到匿名模式。
4. 当前已加载内容和 Vault 笔记不被删除。

### AC-07 阅读历史（问题查询记录）

1. 用户查询一个问题 URL 并成功打开，历史列表新增一条以问题 ID 为键的记录。
2. 用户直接查询一个回答 URL 并成功打开，历史列表记录该回答所属的问题 ID，不保存回答 ID。
3. 用户查询同一问题下的不同回答，历史列表仍只有一条该问题记录，只更新时间并置顶。
4. 用户在阅读面板中切换回答时，不新增或更新历史。
5. 用户点击工具栏“历史列表”，Popover 按时间倒序展示问题标题。
6. 用户点击某个问题标题后，Popover 关闭，阅读器以问题模式打开该问题并加载第一批回答。
7. 删除单条、清空历史或关闭历史记录均不会删除 Vault 中的笔记和附件。

### AC-08 每日热榜

1. 用户通过工具栏或命令打开每日热榜，Popover 展示带名次和标题的热门问题列表。
2. 五分钟内重复打开复用内存缓存；点击刷新会重新请求榜单。
3. 只浏览或刷新榜单不会新增查询历史，也不会创建 Vault 文件。
4. 用户点击某个条目后，Popover 关闭，阅读器以问题模式打开该问题；加载成功后历史新增或置顶该问题。
5. 未登录且接口拒绝访问时，Popover 明确提示先在插件设置中登录，并允许重试。
6. 加载、空列表、失败、Escape、外部点击和焦点返回行为符合 UI 规范。

## 9. 待验证事项

- 知乎不同账号状态下公开问题、单篇回答和 feed API 的稳定性及限流策略。
- 二维码登录 API 在 Obsidian 桌面端与移动端的可用性。
- 知乎公式、视频、盐选/付费内容和折叠回答的实际响应格式。
- Obsidian 移动端对请求头、Cookie、剪贴板和大图下载的限制。
- 单篇回答切换时，当前会话应保留多少篇已读回答，以及点击“下一回答”的可接受等待时间。
- 知乎热榜接口的登录要求、排序语义和字段结构是否会随客户端版本变化。

这些事项影响实现方式，但不改变“临时阅读不污染 Vault，主动选择后再沉淀”的产品原则。
